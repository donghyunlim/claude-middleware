# /breakdown — 큰 프로젝트, 자동으로 쪼개서 동시에 실행합니다

> 프로젝트가 너무 커서 어디서부터 시작해야 할지 모르겠을 때.
> `/breakdown`은 큰 요청을 **자동으로 작은 단위로 쪼개고**, 각 단위에 적합한 AI를 배정해서 **동시에 실행**합니다.

---

## 이런 분에게 추천합니다

- **여러 파일, 모듈, 서비스에 걸친 대규모 작업**을 맡기는 분
- 작업이 복잡해서 **무엇부터 해야 할지 정리가 안 되는** 분
- 독립적인 하위 작업을 **병렬로 빠르게 처리**하고 싶은 분
- 개발, 테스트, 문서화 등 **여러 역할이 동시에 필요한** 프로젝트를 진행하는 분

---

## 어떻게 동작하나요?

### 1단계: 복잡한 요청을 자동으로 작은 조각으로 쪼갭니다

큰 문제를 있는 그대로 풀면 정확도가 매우 낮습니다.
하지만 작은 단위로 쪼개면 극적으로 올라갑니다.

이 원리가 얼마나 강력한지 보여주는 수치입니다:

| 문제 유형 | 분해 전 | 분해 후 | 변화 | 검증 지표 |
|---|---|---|---|---|
| 복잡한 명령어 조합 | 16% | 99.7% | **+84%p** | SCAN 정확도 |
| 다단계 추론 | 4% | 74% | **+70%p** | Game of 24 성공률 |
| 다중 단계 질문 응답 | 17.6% | 60% | **+42.4%p** | Bamboogle 정확도 |

> 📎 Zhou et al. "Least-to-Most Prompting" (2022) arXiv:2205.10625 · Yao et al. "Tree of Thoughts" (2023) arXiv:2305.10601 · Press et al. "Self-Ask" (2022) arXiv:2210.03350

`/breakdown`은 이 분해 원리를 **프로젝트 규모**로 확장합니다.
여러분의 요청을 재귀적으로 분해하여,
각각이 독립적으로 실행 가능한 하위 작업으로 만듭니다.

### 2단계: 실행 전에 상세한 계획서를 작성합니다

무작정 실행하지 않습니다.
먼저 **모든 하위 작업의 목표, 범위, 의존 관계, 성공 기준**을
명시한 실행 계획서를 작성합니다.

계획서에 포함되는 내용:

| 항목 | 설명 |
|---|---|
| 하위 작업 목록 | 각 작업의 명확한 목표와 범위 |
| 의존 관계 그래프 | 어떤 작업이 어떤 작업에 의존하는지 |
| 실행 순서 | 병렬 가능한 작업과 순차 필요 작업 구분 |
| AI 배정 계획 | 각 작업에 어떤 전문 AI를 붙일지 |
| 성공 기준 | 각 하위 작업의 완료 조건 |

모든 AI가 같은 맥락과 목표를 공유하기 때문에
"각자 다른 방향으로 가는" 문제가 발생하지 않습니다.

### 3단계: 실행 전략을 자동으로 선택합니다

분해 결과를 보고 4가지 실행 전략 중 최적을 자동 선택합니다:

| 상황 | 전략 | 설명 |
|---|---|---|
| 분해 불필요 | 단일 실행 | AI 하나가 바로 처리 |
| 순서가 중요 | 순차 실행 | A 완료 후 B 시작 |
| 독립 작업이 많음 | 병렬 실행 | A와 B를 동시에 |
| 혼합 상황 | 하이브리드 | 일부 병렬 + 일부 순차 |

불필요한 복잡성을 추가하지 않습니다.
단순한 요청이면 분해 없이 바로 실행합니다.

### 4단계: 각 조각에 가장 적합한 AI를 자동 배정합니다

코드 작성, 테스트, 문서화 — 각 하위 작업은 성격이 다릅니다.

`/breakdown`은 각 작업의 특성을 분석해서
**가장 적합한 AI 전문가를 자동으로 배정**합니다.

| 하위 작업 유형 | 배정되는 AI |
|---|---|
| 코드 작성 | 코드 전문 AI |
| 테스트 작성 | 테스트 전문 AI |
| 문서화 | 문서 전문 AI |
| 설정/인프라 | 설정 전문 AI |

범용 AI 하나가 전부 처리하는 것보다
각 영역의 전문가가 담당하는 것이 품질이 높습니다.

### 5단계: 독립적인 작업은 동시에 실행합니다

하위 작업 간의 의존 관계를 자동으로 파악합니다.

- A가 끝나야 B를 할 수 있다면 -> **순서대로** 실행
- A와 B가 서로 독립적이라면 -> **동시에** 실행

순차 실행 대비 전체 소요 시간이 크게 단축됩니다.

### 6단계: 작업 중 실패하면 자동으로 에스컬레이션합니다

하위 작업 중 하나가 실패해도 무한 재시도하지 않습니다.

| 상황 | 대응 |
|---|---|
| 자동 수정 가능한 오류 | 재시도 후 진행 |
| 의존 관계에 영향 | 관련 작업 중단 |
| 사용자 판단 필요 | 에스컬레이션 (사용자에게 알림) |
| 핵심 작업 실패 | 전체 중단 + 상황 보고 |

**무한 루프 없이 안전하게** 처리됩니다.

## 이전보다 뭐가 좋아지나요?

| 상황 | 기존 AI | `/breakdown` 사용 시 | 개선 |
|---|---|---|---|
| 복잡한 문제 | 통째로 풀려다 실패 | 작은 조각으로 나눠 해결 | **+84%p (SCAN 정확도)** |
| 다단계 추론 | 중간에 길을 잃음 | 단계별 분해 후 정복 | **+70%p (Game of 24 성공률)** |
| 대규모 프로젝트 | 한 번에 하나씩 순차 처리 | 독립 작업 동시 실행 | **실행 시간 단축** |
| AI 역할 배정 | 범용 AI가 전부 처리 | 작업별 전문 AI 배정 | **작업 품질 향상** |
| 실행 중 실패 | 무한 재시도 또는 포기 | 자동 에스컬레이션 | **안전한 실패 처리** |
| 팀 조율 | 각자 다른 방향으로 진행 | 공유 계획서 기반 실행 | **방향 일관성** |

## 사용 예시

```
/breakdown "결제 시스템에 구독 기능 추가해줘 — API, DB, 프론트, 테스트 전부"
```

자동으로 하위 작업 생성:
1. DB 스키마 설계 (독립)
2. API 엔드포인트 구현 (DB 스키마 의존)
3. 프론트엔드 UI 구현 (API 의존)
4. 테스트 작성 (API + 프론트 의존)

1번은 즉시 시작, 2~4번은 의존 관계에 따라 순차/병렬 실행.

```
/breakdown "기존 모놀리스를 마이크로서비스로 분리해줘"
```

서비스 경계 식별, 각 서비스 분리, API 게이트웨이 설정,
데이터 마이그레이션, 통합 테스트를 자동으로 분해하고 실행합니다.

## `/ha` 계열과 `/breakdown`의 차이

| 관점 | `/ha` 계열 | `/breakdown` |
|---|---|---|
| 접근 방식 | 추론 -> 질문 -> 실행 | 분해 -> 배정 -> 동시 실행 |
| 초점 | 단일 작업의 추론 품질 | 복잡 작업의 구조 분해 |
| 질문 대상 | 사용자에게 질문 | 작업을 하위 작업으로 분해 |
| 실행 구조 | 단일 실행 | 의존 관계 기반 병렬 실행 |
| AI 배정 | 고정 | 작업별 전문가 자동 배정 |
| 최적 규모 | 단일~중간 | **대규모 다중 컴포넌트** |

## 이 명령어를 쓰면 안 되는 경우

- **단일 파일, 단일 함수 수준의 작업**
  — 분해 자체가 불필요합니다. `/haq`를 사용하세요.

- **무엇을 해야 하는지 자체가 불확실한 탐색적 작업**
  — 먼저 `/haqqq`로 요구사항을 명확히 한 뒤 `/breakdown`을 사용하세요.

- **빠른 반복이 필요한 경우**
  — 분해 + 계획 단계에 시간이 소요됩니다. `/haq`가 적합합니다.
